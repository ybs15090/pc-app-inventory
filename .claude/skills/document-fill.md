# 补全文档 Skill

## 触发条件

当用户说"补全"某个 Markdown 软件文档时触发。目标文件通常是一个只有目录骨架和标题的空壳 `.md`，需要将其扩充为完整的软件说明文档。

典型触发词：**补全**、**完善**、**写完**、**填充** + 文件路径。

## 工作流

### 1. 读取目标文件

读取用户指定的 `.md` 文件，了解已有内容（通常是 TOC + 标题 + 几条链接）。

### 2. 读取规范与参考

- 读取仓库根目录的 `CLAUDE.md`，确认文档规范
- 读取同目录下的 `README.md`，了解该文件夹的定位与软件清单

### 3. 选取风格参照

在同目录或同级目录中找一份已完成的优秀文档作为风格参照，例如 `General/Essential/file-search/ripgrep.md`。

### 4. 检查媒体资源

在 `.media/svg/` 和 `.media/png/` 中搜索该软件是否有对应图标。如果有，按 CLAUDE.md 中的图标用法嵌入标题。

### 5. 研究软件（按需）

如果对该软件了解不够充分，使用 `WebSearch` 或 `WebFetch` 获取官方文档、安装方式、使用示例等最新信息。

### 6. 撰写文档

按照以下结构补全（与 CLAUDE.md 保持一致）：

```markdown
# 目录 <!-- omit in toc -->
- [标题](#标题)
  - [安装](#安装)
  - [快速入门/使用方法](#快速入门使用方法)
    - [子章节（按需）](#子章节按需)
  - [相关链接](#相关链接)

# 标题（含图标，如有）

一句话简介。

## 安装

按平台分小节（Windows / macOS / Linux）。

**每个平台的安装章节应同时提供以下方式：**
- **包管理器**（首选）：winget/Scoop/Chocolatey（Windows）、Homebrew（macOS）、apt/dnf/pacman（Linux）
- **本地安装包**（备选）：`.msi`/`.exe`（Windows）、`.dmg`（macOS）、`.deb`/`.rpm`（Linux），或从官网/GitHub Releases 下载的二进制文件

Linux 平台应始终提及通过 `.deb` / `.rpm` 等本地安装包作为备选方案，因为部分场景下用户无法或不希望使用外部包管理器。

## 快速入门 / 使用方法

核心功能说明，优先使用表格和代码块呈现，便于速查。

## 相关链接

- 官网
- GitHub
- 文档/教程
```

### 写作规范

- **语言**：中文撰写
- **平台标注**：如软件仅限特定平台，在简介中明确说明，并在文内链接跨平台替代方案
- **表格优先**：命令速查、语法对照等内容优先使用表格
- **代码块**：命令示例使用 bash 代码块
- **图标路径**：从目标文件所在位置计算 `.media/` 的相对路径

### 7. 更新目录索引

补全完成后，检查同目录的 `README.md`，将新文件加入清单列表（如尚未列出）。每项附带一句话说明其用途。

### 8. 报告

向用户简要说明补全的内容板块，列出新增的章节标题。

## 注意事项

- 优先保留用户已有的任何内容（TOC 结构、已有链接等），只在其基础上扩充
- 如果用户已经移动了文件（平台变更），注意更新相关的 README 索引和交叉引用
- 非跨平台软件应放在对应平台目录下（`Windows/`、`Linux/`、`Mac/`），而非 `General/`
